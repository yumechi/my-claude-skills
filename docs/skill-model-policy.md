# スキルのモデル割り当てポリシー

## 目的

スキル実行のコストを下げるため、各スキルを メイン会話とは独立したサブエージェント として、処理の性質に見合った安いモデル で実行する。メインエージェント（通常は Opus）のモデル・コンテキストには影響を与えない。

## 仕組み（Claude Code の仕様）

SKILL.md のフロントマターで以下を指定する。

| フィールド | 値 | 役割 |
|---|---|---|
| `context` | `fork` | スキルを独立したサブエージェントコンテキストで実行する。メイン会話のモデル・コンテキストは無傷。 |
| `agent` | `general-purpose` | fork 時に使うサブエージェント種別。ファイル書き込みや Bash が必要なため汎用エージェントを指定。 |
| `model` | `haiku` / `opus[1m]` 等 | そのサブエージェントで使うモデル。 |

### 補足: `model:` 単体と `context: fork` の違い

- `model:` のみ（fork なし）: 会話コンテキストは保持されたままモデルだけが切り替わり、次プロンプトでセッションモデルに戻る。ただし「メインのターン自体が一時的に安いモデルになる」点は避けられない。
- `context: fork`: メインエージェントは無傷のまま、独立サブエージェントだけが安いモデルで動く。本リポジトリでは fork を採用 する。

### fork の制約

fork されたサブエージェントは:

- 会話履歴を見られない — リポジトリ／git の状態だけを入力に動く。
- 実行中にユーザーへ確認を取れない — 自律実行して結果を返すだけ。

このため、スキルは「会話の文脈に依存せず、リポジトリの状態から自己完結で判断できる」設計にする必要がある。

## スキルの2種別

### 自律系

処理を最後まで完遂し、結果（レポートや成果物の URL）を返す。読み取り中心、または定義済みアクションの実行が目的のもの。

### 提案→適用系

書き込み前にユーザー確認が必要なスキル。fork 下では確認が取れないため、次の流れにする。

1. fork されたサブエージェント（安いモデル）が 検出・下書き生成までを行い、ファイル書き込みやコミットはしない。
2. 末尾に明示的な「適用手順」ブロック（書き込み先・内容、または実行コマンド）を付けて 提案を返す。
3. メインエージェントがそれをユーザーに提示し、承認後に適用 する。

これにより、重いトークン処理は安いモデルで行いつつ、確認ステップを維持する。

## モデル割り当て

| スキル | model | 種別 |
|---|---|---|
| `clean-branch` | `haiku` | 自律 |
| `todo-scan` | `haiku` | 自律（読み取り） |
| `create-pr` | `haiku` | 自律 |
| `add-gitignore` | `haiku` | 提案→適用 |
| `quickcommit` | `haiku` | 提案→適用 |
| `dependency-audit` | `opus[1m]` | 自律（読み取り） |
| `check-public-repository` | `opus[1m]` | 自律（読み取り） |
| `update-docs` | `opus[1m]` | 提案→適用 |

### 今後追加するスキルのデフォルト

明示的に指定しない限り、新規スキルは `sonnet[1m]` をデフォルトとする。タスクの性質に応じて上下に調整する（機械的な作業は `haiku`、判断・品質が重要なものは `opus[1m]`）。

## フロントマターのテンプレート

### 自律系

```yaml
---
name: <skill-name>
description: <説明>
context: fork
agent: general-purpose
model: <haiku | opus[1m] | sonnet[1m]>
---
```

### 提案→適用系

フロントマターは同じ。本文に「ファイルを書き込まず、提案として返す」旨と「適用手順」セクションを明記する。

## デプロイ

編集対象は `skills/` 配下のみ。`./copy.sh`（または `copy.ps1`）で `~/.claude/skills/` に反映する。本ドキュメント（`docs/`）は配布対象外。
